function build_buf_obj(bufnr, visible)
  local buf = {
    bufnr = bufnr,
    path = vim.api.nvim_buf_get_name(0),
    modified = vim.bo[bufnr].modified,
    readonly = vim.bo[bufnr].readonly,
  }
  if bufnr == state.current_bufnr then
    buf.visibility = 2
  else
    buf.visibility = 0 -- invisible
    for _, visible_bufnr in ipairs(visible) do
      if visible_bufnr == bufnr then
        buf.visibility = 1
        break
      end
    end
  end
end

OBS: Em lua, #tabela retorna a quantidade de elementos POSICIONAIS (como se fosse array).
     A gente quer saber quantas CHAVES a tabela tem. Nesse caso, o operador # não vai funcionar.
     Opções:
       - Calcular as chaves de uma tabela e ver o tamanho
         - Conseguimos guardar todas as chaves se isso for interessante para depois
         - Alocações possivelmente desnecessárias, visto que normalmente os
           arquivos já têm nomes únicos
       - Iterar pela tabela e parar na segunda volta
         - Nenhuma alocação de tabela, no máximo um inteiro/booleano, nos casos
           mais comuns
         - No caso de que realmente o nó têm mais de um filho, vamos ter que
           iterar de qualquer jeito pela tabela
Eu optei pela segunda opção para evitar alocação de memória nos casos mais comuns.

No caso de achar um nó com apenas 1 filho, graças à propriedade MEGA IMPORTANTE
citada acima, podemos simplesmente parar de coletar os caminhos e seguir
recurisvamente até o final do galho, até chegar na última folha (leaf), que é
uma tabela sem nenhuma chave mas com um valor POSICIONAL na primeira posição,
que em lua é o índice 1. Esse valor é o bufnr do buffer que tem esse nome como o
seu nome único de arquivo.

Depois, fazemos o flat dessa buf_tree com esses caminhos unicamente
identificáveis e seus bufnrs, assim:

{
  [bufnr_1] = <nome único 1>,
  [bufnr_2] = <nome único 2>,
  [bufnr_3] = <nome único 3>,
}

OBS: Usar lógica de truncamento do nome aqui.

Perceba que em lua, quando os índices são inteiros sequenciais começando em 1,
essa tabela "é um array". O que isso pode gerar é que caso tenhamos bufnrs
começando em 1, ao fazer um print dessa tabela, ela pode aparecer como um array,

--------------------------
ESTÁGIO 5: Fazer a tabline
--------------------------

A ideia geral é calcular uma janela visível, quais índices começamos e
terminamos de iterar sobre state.buffers para mostrar. Pode ser que não caibam
todos os buffers, então essa janela se torna bastante útil.

A ideia também é manter o buffer atual centralizado, a menos que estejamos perto
do primeiro ou último buffer.

Quando chegamos no primeiro ou último buffer, não damos a volta na hora de
mostrar a tabline, mas um :Bufnext ou :Bufprevious deve sim dar a volta.

A implementacão foi feita assim:

Para cada state.buffer, é calculado um "entry".
Inicialmente eu não gostei muito dessa ideia, porque os buffers já são uma
estrutura organizada com as informações necessárias para o render. Mas eu refiz
as estruturas num novo formato para mais clareza. Provavelmente é possível
remover isso, mas seguimos. Um entry tem o seguinte formato:

{
  path = //o caminho calculado pela buf_tree (e buf_tree_flattened)
  hl = //o highlight group a ser usado. Depende da visibilidade (atual, visível, invisível)
  width = //o tamanho que essa entry gasta na tabline
  bufnr = //o seu bufnr
}

Munido da lista de entries, calcula-se a janela visível: start_i e end_i.
Estes são inteiros, índices onde devemos buscar buffers em state.buffers/entries
para mostrar.
A ordem das entries é a mesma de state.buffers. Dessa forma, ao reordenar o
state.buffers, reordenaremos as entries.

É feita uma tentativa de manter o current_i mais ou menos centralizado dentro
dessa janela.

Esses índices são calculados de forma a tentar preencher toda a tabline mas não
passar o tamanho total dela, levando em conta o tamanho do nome do arquivo,
incluindo modificadores (readonly, modified) e um padding.
Esse padding hoje está hardcoded na função renderer.render_window. No final há
algo assim: string.format(... %s ...), que vira o entry.path.
Eu odiei hard codar o padding, mas como é um código usado por mim, e não um
plugin, eu deixei assim. Não tenho a flexibilidade de um nvim-cokeline, por
exemplo, que é maravilhoso.

Então iteramos pelos entries visíveis (de start_i até end_i), truncamos nomes
muito grandes e renderizamos.

Existem dois tipos de truncamentos:
1. Nome do arquivo muito grande (opts.max_buffer_name_size)
2. Nome não caberia na tabline

Para o segundo tipo, não truncamos o buffer atual.
Truncamos somente nas pontas também.
